# ADR-001: V0.2 Continuous Action Protocol

## Status

Proposed for V0.2 implementation.

## Context

SharedGround V0.1 proves the shared workspace model with one browser workspace state, typed actions, reducer permissions, Human edits, Agent turns, Activity Log, and Evaluation. The important limitation is the Agent execution boundary:

```text
Browser sends workspace snapshot
API applies Agent actions
API returns complete replacement workspace state
Browser replaces local state
```

This is acceptable for V0.1's click-to-run demo, but it is unsafe for V0.2. If the Human edits Evidence, Claim, or Brief while the Agent request is in flight, the API response may contain an older workspace and overwrite the Human's newer local state.

V0.2 needs weak asynchronous collaboration, not distributed real-time collaboration:

- Human can modify shared objects while Agent is working.
- Agent works in small atomic steps.
- Every Agent step rereads the latest workspace.
- All writes still pass through the local reducer.
- Human changes win over stale Agent writes.
- No multi-user, login, Redis, database, WebSocket, SSE, CRDT, automatic merge, vector database, RAG pipeline, live web search, or multi-agent runtime.

## Decision

Use a browser-owned continuous action protocol:

```text
Browser Workspace Store
  -> Human actions immediately apply through reducer
  -> Agent loop calls /api/agent-step with latest snapshot
  -> API returns AgentTurn / proposed actions only
  -> Browser applies each action against latest local state
  -> Reducer accepts or rejects each action
  -> Agent receives apply results and re-reads latest state
  -> Loop continues, pauses, waits for human, completes, or errors
```

The API must not return an authoritative replacement `WorkspaceState` for normal Agent execution. It returns a validated `AgentTurn` with proposed actions. The browser applies those actions to whichever workspace is current at apply time.

Each existing-object Agent update must include `expectedVersion`. The reducer rejects mismatches with `STALE_OBJECT_VERSION`. Human actions do not require `expectedVersion` and are applied immediately, subject to schema and reference validation.

Add `runId`, `stepId`, and `actionId`:

- `runId` identifies one active run created by Start or by pressing Resume.
- `stepId` identifies one Agent request within a run.
- `actionId` identifies one proposed action and its apply result.

The browser discards Agent responses whose `runId` or `stepId` no longer matches the active run/step. Pause and Reset abort in-flight requests when possible and also invalidate old responses. Pause terminates the current run. Resume never continues the old `runId`; it creates a fresh `runId` from the latest workspace. The UI may still say "Resume", but the technical semantics are "start a new run from current state." Page refresh never restores an in-flight run.

Use canonical actor fields in V0.2 entities:

- `createdBy` is the canonical creator.
- `updatedBy` is the canonical last modifier.
- `addedBy` is V0.1 compatibility only and may be used by migration to backfill `createdBy`.
- New business logic, permissions, selectors, and evaluation must not depend on `addedBy`.
- Removing `addedBy` is deferred to V0.3 or later compatibility cleanup.

Human teammate messages are not double-written. `HumanTeammateMessage` is the authoritative message body. A `SEND_TEAMMATE_MESSAGE` event stores only the message object's ID, a short summary, and small metadata. Acknowledgement is still keyed by the event ID; context building resolves the event to the message object when full text is needed.

Acknowledgement is not a model-emitted action. Do not add `ACKNOWLEDGE_HUMAN_CHANGES` to the `WorkspaceAction` discriminated union. The model declares acknowledgement only in `AgentTurn.acknowledgedHumanEventIds`; after action application, the store persists those IDs through a reducer-backed/system helper and may write a slim system event.

## Alternatives Considered

### Keep V0.1 Full-State Replacement

Rejected. It is simpler, but it directly conflicts with V0.2's requirement that Human may edit while Agent is working. It risks stale server state overwriting newer Human changes.

### WebSocket

Rejected for V0.2. There is no multi-user collaboration, no server push requirement, and no shared server-owned session state. It would add infrastructure without solving the core problem better than local reducer version checks.

### Server-Sent Events

Rejected for V0.2. SSE is useful for streaming server updates to the browser. V0.2 Agent steps are request-response proposals, and the browser remains the state owner.

### Redis Or Backend Queue

Rejected for V0.2. Redis is useful for multi-process coordination and durable queues. SharedGround V0.2 is a single-user local demo and should prove the protocol before adding distributed runtime components.

### Backend Database

Rejected for V0.2. A database improves durability but does not change the Human-Agent conflict protocol. localStorage is sufficient with Markdown size limits and slim event logs.

### CRDT Or Automatic Merge

Rejected for V0.2. SharedGround objects are semantic research objects, not collaborative text buffers. Human authority should be explicit and auditable, so stale Agent writes should be rejected rather than silently merged.

## Consequences

Positive:

- Human edits cannot be overwritten by old API state.
- Conflict handling is local, deterministic, testable, and auditable.
- V0.1 reducer and permission model remain central.
- No new infrastructure is required.
- Evaluation can measure stale write rejection and Human change response.

Costs:

- Store logic becomes more complex because it owns run/step lifecycle.
- API tests must change from "returns state" to "returns actions".
- Agent context must include previous apply results so the Agent knows which actions succeeded.
- Mock Agent must adapt to step-based behavior.

## Required Invariants

- All state mutations go through the reducer or reducer-backed store helpers.
- The Agent never replaces the entire workspace.
- Agent existing-object updates include `expectedVersion`.
- `Source.content` for uploaded Markdown is immutable.
- Events are slim and never store full source content.
- `ActionApplyResult` is produced for every attempted action.
- `acknowledgedHumanEventIds` are explicit; a stale-rejected action does not count as handling a Human change.
- Brief freshness is computed from derivation version maps, not stored as a mutable boolean.
- Pause invalidates the current run; Resume creates a new `runId`.
- Page refresh does not recover in-flight runs.
- `createdBy` and `updatedBy` are authoritative actor fields; `addedBy` is compatibility-only.
- Human message text lives in `HumanTeammateMessage`, not duplicated in events.
- Human-change acknowledgement lives on `AgentTurn.acknowledgedHumanEventIds`, not in `WorkspaceAction`.

## Implementation Notes

Recommended phase order:

1. Versioned Core & Slim Events.
2. Action Apply Results.
3. Markdown Sources & Stable Evidence Location.
4. Agent Step Loop & Request Race Protection.
5. Human Messages & Recent Human Changes.
6. Claim / Brief Derivation Discipline.
7. Evaluation V0.2.
8. UI Integration, Docs & Demo Packaging.
